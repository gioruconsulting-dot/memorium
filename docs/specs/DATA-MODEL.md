# Memorium — Data Model

**Database**: SQLite (via Turso)  
**ORM**: Drizzle ORM (optional, can use raw SQL if simpler)  
**Migration strategy**: Schema-first, version controlled

---

## Schema Overview

```
users (v2 only — prepared but not used in v1)
  ↓
documents (1 user → many documents)
  ↓
questions (1 document → many questions)
  ↓
study_sessions (tracks each daily session)
  ↓
session_answers (tracks each question answered in a session)
```

## Tables

### 1. `users` (v2 — Auth-ready but unused in v1)

Prepared for future multi-user support. In v1, all data uses `userId = 'default-user'`.

```sql
CREATE TABLE users (
  id TEXT PRIMARY KEY,              -- e.g., 'user_abc123' (Clerk userId in v2)
  email TEXT,                       -- Optional for v1
  created_at INTEGER NOT NULL,      -- Unix timestamp
  last_active_at INTEGER NOT NULL   -- Unix timestamp
);
```

**Fields:**
- `id`: Primary key. In v1, always `'default-user'`. In v2, populated by Clerk.
- `email`: User's email (for v2 notifications/export). Nullable for v1.
- `created_at`: When user first used the app.
- `last_active_at`: Last study session date (for streak calculation).

**v1 behavior**: Insert one row on first use with `id = 'default-user'`.

### 2. `documents`

Stores uploaded documents and metadata.

```sql
CREATE TABLE documents (
  id TEXT PRIMARY KEY,              -- e.g., 'doc_1234567890123'
  user_id TEXT NOT NULL,            -- Foreign key to users.id
  title TEXT NOT NULL,              -- User-provided or filename
  content TEXT NOT NULL,            -- Full text content
  themes TEXT,                      -- Comma-separated tags (e.g., 'Philosophy,Stoicism')
  question_count INTEGER NOT NULL,  -- Total questions generated (always 20)
  created_at INTEGER NOT NULL,      -- Unix timestamp
  
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

CREATE INDEX idx_documents_user_id ON documents(user_id);
CREATE INDEX idx_documents_created_at ON documents(created_at);
```

**Fields:**
- `id`: Generated via `Date.now()` or UUID. Must be unique.
- `user_id`: Always `'default-user'` in v1.
- `title`: Display name (e.g., "Chapter 3: Stoic Ethics").
- `content`: Full text (stored for reference, not re-processed).
- `themes`: Comma-separated string (e.g., `'Philosophy,Stoicism,Ancient Greece'`). Used for filtering study sessions.
- `question_count`: Always 20 in v1 (might be configurable in v2).
- `created_at`: Upload timestamp.

**Example row:**
```json
{
  "id": "doc_1234567890123",
  "user_id": "default-user",
  "title": "Meditations by Marcus Aurelius - Book 2",
  "content": "Begin each day by telling yourself...",
  "themes": "Philosophy,Stoicism,Self-Reflection",
  "question_count": 20,
  "created_at": 1714521600
}
```

### 3. `questions`

Stores generated questions and spaced repetition state.

```sql
CREATE TABLE questions (
  id TEXT PRIMARY KEY,              -- e.g., 'q_1234567890123'
  document_id TEXT NOT NULL,        -- Foreign key to documents.id
  user_id TEXT NOT NULL,            -- Denormalized for easy querying
  
  -- Question content
  question_text TEXT NOT NULL,      -- The question
  question_type TEXT NOT NULL,      -- 'recall' | 'application' | 'connection'
  answer_text TEXT NOT NULL,        -- Model answer
  explanation TEXT,                 -- Why this answer is correct
  source_reference TEXT,            -- Quote from original content
  
  -- Spaced repetition state
  next_review_at INTEGER NOT NULL,  -- Unix timestamp (when question is due)
  review_count INTEGER DEFAULT 0,   -- Total times reviewed
  correct_count INTEGER DEFAULT 0,  -- Times graded Easy or Hard
  incorrect_count INTEGER DEFAULT 0,-- Times graded Forgot
  correct_streak INTEGER DEFAULT 0, -- Consecutive Easy/Hard (resets on Forgot)
  hard_count INTEGER DEFAULT 0,     -- Times graded Hard (vs Easy)
  current_interval_days INTEGER DEFAULT 1, -- Current spacing interval
  
  -- Metadata
  created_at INTEGER NOT NULL,      -- Unix timestamp
  
  FOREIGN KEY (document_id) REFERENCES documents(id) ON DELETE CASCADE,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

CREATE INDEX idx_questions_document_id ON questions(document_id);
CREATE INDEX idx_questions_user_id ON questions(user_id);
CREATE INDEX idx_questions_next_review_at ON questions(next_review_at);
CREATE INDEX idx_questions_review_count ON questions(review_count);
```

**Fields:**

**Content:**
- `question_text`: The actual question (e.g., "What did Marcus Aurelius recommend doing each morning?")
- `question_type`: One of `['recall', 'application', 'connection']`. Generated by Claude.
- `answer_text`: The correct answer (e.g., "Begin each day by telling yourself: Today I shall be meeting...")
- `explanation`: Why this answer is correct + additional context.
- `source_reference`: Direct quote from the document that supports the answer.

**Spaced Repetition State:**
- `next_review_at`: Unix timestamp. Question appears in session when `next_review_at <= current_time`.
- `review_count`: Total number of times user has reviewed this question.
- `correct_count`: Number of times graded Easy or Hard.
- `incorrect_count`: Number of times graded Forgot.
- `correct_streak`: Consecutive Easy/Hard grades. Resets to 0 on Forgot.
- `hard_count`: Number of times graded Hard (tracks difficulty).
- `current_interval_days`: Current spacing interval (1, 3, 7, 14, 30, 60, 90, 180).

**Mastery logic:**
- Questions never retire, but intervals can extend up to 180 days.
- High `correct_streak` + low `incorrect_count` = longer intervals.

**Example row:**
```json
{
  "id": "q_1234567890124",
  "document_id": "doc_1234567890123",
  "user_id": "default-user",
  "question_text": "What daily practice did Marcus Aurelius recommend for self-preparation?",
  "question_type": "recall",
  "answer_text": "Begin each day by reminding yourself that you'll encounter difficult people and situations, but you can choose how to respond with virtue.",
  "explanation": "This is from Book 2 of Meditations, where Marcus prepares himself mentally for the day ahead by acknowledging reality while maintaining agency over his responses.",
  "source_reference": "Begin each day by telling yourself: Today I shall be meeting with interference, ingratitude, insolence...",
  "next_review_at": 1714608000,
  "review_count": 3,
  "correct_count": 3,
  "incorrect_count": 0,
  "correct_streak": 3,
  "hard_count": 1,
  "current_interval_days": 7,
  "created_at": 1714521600
}
```

### 4. `study_sessions`

Tracks each daily study session (for analytics and streak calculation).

```sql
CREATE TABLE study_sessions (
  id TEXT PRIMARY KEY,              -- e.g., 'session_1234567890123'
  user_id TEXT NOT NULL,            -- Foreign key to users.id
  
  started_at INTEGER NOT NULL,      -- Unix timestamp
  completed_at INTEGER,             -- Unix timestamp (null if abandoned)
  
  questions_shown INTEGER NOT NULL, -- Total questions in session (up to 15)
  questions_answered INTEGER DEFAULT 0, -- Questions user actually graded
  
  correct_count INTEGER DEFAULT 0,  -- Graded Easy or Hard
  incorrect_count INTEGER DEFAULT 0,-- Graded Forgot
  skipped_count INTEGER DEFAULT 0,  -- Clicked Skip
  
  duration_seconds INTEGER,         -- Actual time spent (completed_at - started_at)
  
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

CREATE INDEX idx_study_sessions_user_id ON study_sessions(user_id);
CREATE INDEX idx_study_sessions_started_at ON study_sessions(started_at);
```

**Fields:**
- `started_at`: When user clicked "Start Session".
- `completed_at`: When user finished the last question. Null if abandoned.
- `questions_shown`: Total questions loaded (e.g., 15 or fewer if not enough due).
- `questions_answered`: Number graded (excludes skips).
- `correct_count`: Graded Easy or Hard.
- `incorrect_count`: Graded Forgot.
- `skipped_count`: Clicked Skip button.
- `duration_seconds`: Actual time spent (for analytics).

**Usage:**
- **Streak calculation**: Count consecutive days with `completed_at IS NOT NULL`.
- **Analytics**: Average accuracy, session length, completion rate.

### 5. `session_answers`

Tracks each individual question answered within a session (optional for v1, useful for analytics).

```sql
CREATE TABLE session_answers (
  id TEXT PRIMARY KEY,              -- e.g., 'answer_1234567890123'
  session_id TEXT NOT NULL,         -- Foreign key to study_sessions.id
  question_id TEXT NOT NULL,        -- Foreign key to questions.id
  user_id TEXT NOT NULL,            -- Denormalized for querying
  
  user_attempt TEXT,                -- What the user typed before revealing answer
  grade TEXT NOT NULL,              -- 'easy' | 'hard' | 'forgot' | 'skipped'
  elaboration_note TEXT,            -- Optional "why this matters" note
  
  answered_at INTEGER NOT NULL,     -- Unix timestamp
  
  FOREIGN KEY (session_id) REFERENCES study_sessions(id) ON DELETE CASCADE,
  FOREIGN KEY (question_id) REFERENCES questions(id) ON DELETE CASCADE,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

CREATE INDEX idx_session_answers_session_id ON session_answers(session_id);
CREATE INDEX idx_session_answers_question_id ON session_answers(question_id);
```

**Fields:**
- `user_attempt`: Text the user typed before clicking "Reveal Answer".
- `grade`: One of `['easy', 'hard', 'forgot', 'skipped']`.
- `elaboration_note`: Optional note from "why does this matter" prompt.
- `answered_at`: When question was graded.

**Usage:**
- Track what user actually typed (for future "self-grading" features).
- Store elaboration notes (potentially useful for review later).
- Detailed session analytics (time per question, etc.).

**v1 scope decision**: This table is **optional** for v1. You can defer it and just update `questions` table directly when grading. Add this in v2 if you want detailed analytics.

## Relationships

```
users (1) ──────────── (many) documents
                            │
                            └──────── (many) questions
                                          │
users (1) ──────────── (many) study_sessions
                            │
                            └──────── (many) session_answers
                                          │
                                          └──────── (references) questions
```

**Cascade delete rules:**
- Delete user → deletes all their documents, questions, sessions, answers
- Delete document → deletes all its questions
- Delete session → deletes all its answers

## Key Queries

### 1. Build Daily Session (Smart Scheduling)

```sql
-- Get up to 15 questions for today's session
SELECT * FROM questions
WHERE user_id = 'default-user'
  AND next_review_at <= ? -- current timestamp
ORDER BY 
  -- Priority 1: Due questions (earliest first)
  CASE WHEN next_review_at <= ? THEN next_review_at ELSE 9999999999 END,
  -- Priority 2: High-risk questions (most incorrect)
  (incorrect_count * 3 + hard_count * 2 - correct_count - correct_streak * 2) DESC,
  -- Priority 3: Older questions (interleaving)
  created_at ASC
LIMIT 15;
```

**Logic:**
1. Filter: due today (`next_review_at <= now`)
2. Sort: due → risky → old
3. Limit: 15 (or fewer if not enough available)

### 2. Grade Question (Update Spaced Repetition State)

```sql
-- After grading, update question state
UPDATE questions
SET 
  review_count = review_count + 1,
  correct_count = correct_count + ?,     -- +1 if easy/hard, +0 if forgot
  incorrect_count = incorrect_count + ?, -- +1 if forgot, +0 if easy/hard
  correct_streak = ?,                    -- streak + 1 if easy/hard, 0 if forgot
  hard_count = hard_count + ?,           -- +1 if hard, +0 if easy/forgot
  current_interval_days = ?,             -- calculated based on grade
  next_review_at = ?                     -- now + interval
WHERE id = ?;
```

**Interval calculation logic** (in API route):
```javascript
function calculateNextInterval(question, grade) {
  const intervals = [1, 3, 7, 14, 30, 60, 90, 180]; // days
  
  if (grade === 'easy') {
    const newStreak = question.correct_streak + 1;
    const intervalIndex = Math.min(newStreak - 1, intervals.length - 1);
    const baseDays = intervals[intervalIndex];
    return Math.min(baseDays * 2, 180); // Cap at 6 months
  }
  
  if (grade === 'hard') {
    const newStreak = question.correct_streak + 1;
    const intervalIndex = Math.min(newStreak - 1, intervals.length - 1);
    const baseDays = intervals[intervalIndex];
    return Math.min(baseDays, 3); // Keep it soon (1-3 days)
  }
  
  if (grade === 'forgot') {
    return 1; // Tomorrow
  }
}
```

### 3. Get Progress Stats

```sql
-- Total questions
SELECT COUNT(*) FROM questions WHERE user_id = 'default-user';

-- Questions with high mastery (streak ≥ 5)
SELECT COUNT(*) FROM questions 
WHERE user_id = 'default-user' 
  AND correct_streak >= 5;

-- Overall accuracy
SELECT 
  SUM(correct_count) as total_correct,
  SUM(incorrect_count) as total_incorrect,
  ROUND(SUM(correct_count) * 100.0 / (SUM(correct_count) + SUM(incorrect_count)), 1) as accuracy_pct
FROM questions
WHERE user_id = 'default-user';

-- Study streak (consecutive days)
SELECT COUNT(DISTINCT DATE(started_at, 'unixepoch')) as streak
FROM study_sessions
WHERE user_id = 'default-user'
  AND completed_at IS NOT NULL
  AND DATE(started_at, 'unixepoch') >= DATE('now', '-7 days');
```

### 4. Get All Documents (Library View)

```sql
SELECT 
  id,
  title,
  themes,
  question_count,
  created_at
FROM documents
WHERE user_id = 'default-user'
ORDER BY created_at DESC;
```

### 5. Delete Document + Cascade Questions

```sql
-- Handled by foreign key constraint (ON DELETE CASCADE)
DELETE FROM documents WHERE id = ?;
-- Automatically deletes all associated questions
```

## Scope Decision: session_answers Table

**Recommendation**: **Defer to v2** unless you want detailed analytics from day 1.

**Why defer:**
- Not required for core functionality (grading updates `questions` table directly)
- Adds complexity (extra INSERT on every question)
- v1 goal is shipping fast, not analytics

**When to add:**
- v2 if you want to analyze:
  - Time spent per question
  - What users actually typed vs. correct answer
  - Session abandonment patterns
  - Elaboration notes review
